船舶态势与轨迹回放技术方案

本方案覆盖原始轨迹、固定抽稀轨迹、船舶时空索引、实时最新船位缓存、单船/多船/全域回放、 以及态势密度和统计接口。唯一优先级是船只加载速度：首屏只走必要链路，默认轨迹查询走固定抽稀表， 统计、候选、分析、日志和调试能力都不能抢占船只加载链路。

存储 ClickHouse 原始表、固定抽稀表、时空索引表
服务 Spring Boot + ClickHouse HTTP 查询
缓存 RealtimeService 维护最新船位和数据库统计
主链路 配置加载后立即请求实时船位

文档结论

当前技术选型

  • tb_ais_track_raw 保存原始轨迹，TTL 一年。
  • tb_ais_track_thin 保存 60/300/1800 秒固定抽稀点。
  • tb_ship_bucket_index 为多船框选提供候选船索引。
  • 单船轨迹默认读抽稀表；只有单船 samplingMode=raw 会读原始表。

性能红线

  • 首屏先加载配置和实时船位，不把统计和辅助分析串到首屏前面。
  • 候选船查询只走索引表，不直接在原始表按 bbox 聚合。
  • 全域回放默认只给位置帧，不绘制整段全域轨迹线。
  • 新增日志、监控、报表都必须证明不拖慢船只加载。

总体架构

数据流

1. 原始写入 AIS、雷达、北斗及融合源写入 tb_ais_track_raw
2. 抽稀与索引 写入链路维护 tb_ais_track_thin 和 tb_ship_bucket_index
3. API 查询 实时缓存、回放、候选船、统计、密度分析分流
4. 前端渲染 地图 marker 优先渲染，统计和热力按模式延后加载

建表与表结构

当前仓库内的 DDL 只保留三张运行时表。抽稀和索引的写入链路在当前代码里是“读已存在表”， 并没有把物化视图 DDL 一并放进仓库。

原始轨迹表

CREATE TABLE IF NOT EXISTS tb_ais_track_raw
(
  ship_serial_no LowCardinality(String),
  longitude_wgs Float32 CODEC(Gorilla),
  latitude_wgs Float32 CODEC(Gorilla),
  ground_speed Float32 CODEC(Gorilla),
  ground_course Float32 CODEC(Gorilla),
  event_time DateTime('Asia/Shanghai') CODEC(Delta, ZSTD),
  type UInt8
)
ENGINE = MergeTree
PARTITION BY toYYYYMM(event_time)
ORDER BY (ship_serial_no, event_time)
TTL event_time + toIntervalYear(1)
SETTINGS index_granularity = 8192;

固定抽稀轨迹表

CREATE TABLE IF NOT EXISTS tb_ais_track_thin
(
  ship_serial_no LowCardinality(String),
  bucket_size UInt32,
  bucket_start DateTime('Asia/Shanghai') CODEC(Delta, ZSTD),
  event_time DateTime('Asia/Shanghai') CODEC(Delta, ZSTD),
  longitude_wgs Float32 CODEC(Gorilla),
  latitude_wgs Float32 CODEC(Gorilla),
  ground_speed Float32 CODEC(Gorilla),
  ground_course Float32 CODEC(Gorilla),
  type UInt8,
  grid_05_lng Int32 MATERIALIZED toInt32(floor((longitude_wgs + 180) / 0.05)),
  grid_05_lat Int32 MATERIALIZED toInt32(floor((latitude_wgs + 90) / 0.05)),
  grid_5_lng Int32 MATERIALIZED toInt32(floor((longitude_wgs + 180) / 0.5)),
  grid_5_lat Int32 MATERIALIZED toInt32(floor((latitude_wgs + 90) / 0.5)),
  dedup_version UInt32 MATERIALIZED toUInt32(4294967295 - toUnixTimestamp(event_time))
)
ENGINE = ReplacingMergeTree(dedup_version)
PARTITION BY toYYYYMM(bucket_start)
ORDER BY (ship_serial_no, bucket_size, bucket_start)
TTL bucket_start + toIntervalYear(1)
SETTINGS index_granularity = 8192;

船舶时空索引表

CREATE TABLE IF NOT EXISTS tb_ship_bucket_index
(
  grid_05_lng Int32,
  grid_05_lat Int32,
  grid_5_lng Int32,
  grid_5_lat Int32,
  ship_serial_no LowCardinality(String),
  bucket_start DateTime('Asia/Shanghai'),
  min_lng AggregateFunction(min, Float32),
  max_lng AggregateFunction(max, Float32),
  min_lat AggregateFunction(min, Float32),
  max_lat AggregateFunction(max, Float32),
  point_count AggregateFunction(count)
)
ENGINE = AggregatingMergeTree
PARTITION BY toYYYYMM(bucket_start)
ORDER BY (grid_05_lng, grid_05_lat, bucket_start, ship_serial_no)
TTL bucket_start + toIntervalYear(1)
SETTINGS index_granularity = 8192;

表设计说明

设计点 说明
原始表排序键 (ship_serial_no, event_time) 适合单船轨迹和单船点数统计按船号裁剪。
抽稀表排序键 (ship_serial_no, bucket_size, bucket_start) 适合按船号与固定桶查询轨迹。
索引表排序键 细网格优先，随后时间桶和船号，适合 bbox 候选船查询先缩小读取范围。
去重版本 同船同粒度同桶重复写入时，用 dedup_version 保留更早事件点。
聚合状态 索引表只保存 bbox 判定和点数所需状态，候选阶段不回原始点明细。

查询原则

抽稀策略

模式 数据源 规则
auto tb_ais_track_thin zoom <= 7 取 1800 秒，8-10 取 300 秒，zoom >= 11 取 60 秒。
manual tb_ais_track_thin 前端传桶秒数，服务端归一到 60、300、1800 中最近粒度。
raw tb_ais_track_raw 仅单船 只有 mode=single 会命中原始表；多船和全域会回退到固定抽稀路径。

通用约束

  • 所有时间窗口采用左闭右开过滤：event_time >= start 且 event_time < end。
  • 抽稀表按桶边界过滤时向前回退一个桶，再以真实 event_time 二次过滤，避免窗口起点落在桶中间漏点。
  • 单船原始表和单船抽稀表查询使用 PREWHERE ship_serial_no。
  • 多船 ID 列表受 maxMultiShips 限制，候选船分页受 maxCandidateBatchSize 限制。
  • 轨迹明细 SQL 不写业务 LIMIT；返回规模依靠抽稀粒度、时间窗、船舶数量和候选分页控制。
  • 热力密度必须携带 bbox；统计类接口允许独立调用，但前端应延后发起。
  • stepMinutes 只由前端用于拆分态势分析时间片，后端密度 SQL 不按该参数分桶。
  • 信号源类型由 type 映射；AIS 视为 type IN (1,4,5,7)。
  • /api/tracks/multi 当前控制器没有把 body 里的 bbox 传给仓库，轨迹返回的是所选船在时间窗内的点，不做框选截断。

功能查询逻辑与 SQL

下列 SQL 按当前仓库查询形态整理。占位参数写法沿用 ClickHouse HTTP 参数： {shipId: String}、{shipIds: Array(String)}、{bucketSeconds: UInt32}。 时间参数统一通过 parseDateTime64BestEffort(..., 'Asia/Shanghai') 解析。 文档只展示当前接口实际可命中的 SQL 路径：单船可 raw，多船和全域只走抽稀表。

实时最新船位

/api/realtime/latest 先由 RealtimeService 判断内存缓存是否命中。 缓存未命中时，原始表按窗口聚合每艘船的最新位置，并用结果刷新缓存。 

SELECT
  ship_serial_no AS shipId,
  ship_serial_no AS shipName,
  argMax(longitude_wgs, event_time) AS lng,
  argMax(latitude_wgs, event_time) AS lat,
  argMax(ground_speed, event_time) AS speed,
  argMax(ground_course, event_time) AS heading,
  argMax(if(type IN (1, 4, 5, 7), toUInt8(1), toUInt8(0)), event_time) AS isAis,
  argMax(type, event_time) AS shipType,
  toString(max(event_time)) AS time
FROM tb_ais_track_raw
WHERE event_time >= parseDateTime64BestEffort({start: String}, 3, 'Asia/Shanghai')
  AND event_time < parseDateTime64BestEffort({end: String}, 3, 'Asia/Shanghai')
GROUP BY ship_serial_no
ORDER BY ship_serial_no ASC
LIMIT {limit: UInt32};

单船轨迹

默认查询固定抽稀表；窗口先按 bucket_start 裁剪，再按实际点时间过滤。 原始模式改查原始表，不做在线桶聚合。 

-- auto/manual 固定抽稀路径
SELECT
  {shipId: String} AS shipId,
  ship_serial_no AS shipName,
  longitude_wgs AS lng,
  latitude_wgs AS lat,
  ground_speed AS speed,
  ground_course AS heading,
  if(type IN (1, 4, 5, 7), toUInt8(1), toUInt8(0)) AS isAis,
  type AS shipType,
  toString(event_time) AS time
FROM tb_ais_track_thin
PREWHERE ship_serial_no = {shipId: String}
WHERE bucket_size = {bucketSeconds: UInt32}
  AND bucket_start >= parseDateTime64BestEffort({start: String}, 3, 'Asia/Shanghai')
      - toIntervalSecond({bucketSeconds: UInt32})
  AND bucket_start < parseDateTime64BestEffort({end: String}, 3, 'Asia/Shanghai')
  AND event_time >= parseDateTime64BestEffort({start: String}, 3, 'Asia/Shanghai')
  AND event_time < parseDateTime64BestEffort({end: String}, 3, 'Asia/Shanghai')
ORDER BY time ASC;

-- raw 原始点路径
SELECT
  {shipId: String} AS shipId,
  ship_serial_no AS shipName,
  longitude_wgs AS lng,
  latitude_wgs AS lat,
  ground_speed AS speed,
  ground_course AS heading,
  if(type IN (1, 4, 5, 7), toUInt8(1), toUInt8(0)) AS isAis,
  type AS shipType,
  toString(event_time) AS time
FROM tb_ais_track_raw
PREWHERE ship_serial_no = {shipId: String}
WHERE event_time >= parseDateTime64BestEffort({start: String}, 3, 'Asia/Shanghai')
  AND event_time < parseDateTime64BestEffort({end: String}, 3, 'Asia/Shanghai')
ORDER BY time ASC;

多船候选船查询

多船流程为“框选 -> 索引表候选分页 -> 选择 shipIds -> 抽稀表轨迹加载 -> 前端播放窗口渲染”。 候选阶段只读取 tb_ship_bucket_index，返回船号、桶时间范围和点数，不补船型；船型随已选船轨迹返回。 

SELECT
  ship_serial_no AS shipId,
  ship_serial_no AS shipName,
  toString(min(bucket_start)) AS firstTime,
  toString(max(bucket_start)) AS lastTime,
  countMerge(point_count) AS points
FROM tb_ship_bucket_index
WHERE bucket_start >= parseDateTime64BestEffort({start: String}, 3, 'Asia/Shanghai')
  AND bucket_start < parseDateTime64BestEffort({end: String}, 3, 'Asia/Shanghai')
  AND grid_05_lng BETWEEN {westGrid05Lng: Int32} AND {eastGrid05Lng: Int32}
  AND grid_05_lat BETWEEN {southGrid05Lat: Int32} AND {northGrid05Lat: Int32}
GROUP BY ship_serial_no
HAVING maxMerge(max_lng) >= {west: Float64}
  AND minMerge(min_lng) <= {east: Float64}
  AND maxMerge(max_lat) >= {south: Float64}
  AND minMerge(min_lat) <= {north: Float64}
ORDER BY points DESC, shipId ASC
LIMIT {limit: UInt32} OFFSET {offset: UInt32};

多船轨迹

多船轨迹在候选船选择后读取所选船号集合。当前控制器没有把 body 里的 bbox 传到仓库， 因而返回的是所选船在时间窗内的轨迹，而不是框内片段。samplingMode=raw 对多船路径会回退到抽稀表。 

SELECT
  ship_serial_no AS shipId,
  ship_serial_no AS shipName,
  longitude_wgs AS lng,
  latitude_wgs AS lat,
  ground_speed AS speed,
  ground_course AS heading,
  if(type IN (1, 4, 5, 7), toUInt8(1), toUInt8(0)) AS isAis,
  type AS shipType,
  toString(event_time) AS time
FROM tb_ais_track_thin
WHERE bucket_size = {bucketSeconds: UInt32}
  AND ship_serial_no IN {shipIds: Array(String)}
  AND bucket_start >= parseDateTime64BestEffort({start: String}, 3, 'Asia/Shanghai')
      - toIntervalSecond({bucketSeconds: UInt32})
  AND bucket_start < parseDateTime64BestEffort({end: String}, 3, 'Asia/Shanghai')
  AND event_time >= parseDateTime64BestEffort({start: String}, 3, 'Asia/Shanghai')
  AND event_time < parseDateTime64BestEffort({end: String}, 3, 'Asia/Shanghai')
ORDER BY time ASC, shipId ASC;

态势密度热力

热力读取 300 秒抽稀表，按 zoom 选择经纬网格：zoom >= 11 为 0.05 度， zoom 8-10 为 0.1 度，低 zoom 为 0.5 度。前端按 stepMinutes 拆分时间片逐帧请求， 后端每次只返回当前时间片的静态热力点，不在 SQL 中按 stepMinutes 二次分桶。 

SELECT
  floor(longitude_wgs / {grid: Float64}) * {grid: Float64}
      + ({grid: Float64} / 2) AS lng,
  floor(latitude_wgs / {grid: Float64}) * {grid: Float64}
      + ({grid: Float64} / 2) AS lat,
  count() AS count,
  uniqCombined64(ship_serial_no) AS ships
FROM tb_ais_track_thin
WHERE bucket_size = 300
  AND bucket_start >= parseDateTime64BestEffort({start: String}, 3, 'Asia/Shanghai')
      - toIntervalSecond(300)
  AND bucket_start < parseDateTime64BestEffort({end: String}, 3, 'Asia/Shanghai')
  AND event_time >= parseDateTime64BestEffort({start: String}, 3, 'Asia/Shanghai')
  AND event_time < parseDateTime64BestEffort({end: String}, 3, 'Asia/Shanghai')
  AND longitude_wgs BETWEEN {west: Float64} AND {east: Float64}
  AND latitude_wgs BETWEEN {south: Float64} AND {north: Float64}
GROUP BY lng, lat
ORDER BY count DESC
LIMIT {limit: UInt32};

密度格子数量

统计面板只需要格子数时，不返回每个热力格明细。

SELECT count() AS cells
FROM
(
  SELECT
    floor(longitude_wgs / {grid: Float64}) * {grid: Float64}
        + ({grid: Float64} / 2) AS lng,
    floor(latitude_wgs / {grid: Float64}) * {grid: Float64}
        + ({grid: Float64} / 2) AS lat
  FROM tb_ais_track_thin
  WHERE bucket_size = 300
    AND bucket_start >= parseDateTime64BestEffort({start: String}, 3, 'Asia/Shanghai')
        - toIntervalSecond(300)
    AND bucket_start < parseDateTime64BestEffort({end: String}, 3, 'Asia/Shanghai')
    AND event_time >= parseDateTime64BestEffort({start: String}, 3, 'Asia/Shanghai')
    AND event_time < parseDateTime64BestEffort({end: String}, 3, 'Asia/Shanghai')
    -- 可选 bbox 过滤
  GROUP BY lng, lat
) AS density_cells;

全域回放

全域路径固定从 1800 秒抽稀点中按船和 30 分钟桶取位置帧。 这样前端驱动时间轴时只维护当前时刻船位，避免全域轨迹线过载。samplingMode=raw 在全域路径会回退到抽稀表。 

SELECT
  ship_serial_no AS shipId,
  ship_serial_no AS shipName,
  argMax(longitude_wgs, event_time) AS lng,
  argMax(latitude_wgs, event_time) AS lat,
  argMax(ground_speed, event_time) AS speed,
  argMax(ground_course, event_time) AS heading,
  argMax(if(type IN (1, 4, 5, 7), toUInt8(1), toUInt8(0)), event_time) AS isAis,
  argMax(type, event_time) AS shipType,
  toString(max(event_time)) AS time,
  toString(bucket_start) AS bucketStart
FROM tb_ais_track_thin
WHERE bucket_size = 1800
  AND bucket_start >= parseDateTime64BestEffort({start: String}, 3, 'Asia/Shanghai')
      - toIntervalSecond(1800)
  AND bucket_start < parseDateTime64BestEffort({end: String}, 3, 'Asia/Shanghai')
  AND event_time >= parseDateTime64BestEffort({start: String}, 3, 'Asia/Shanghai')
  AND event_time < parseDateTime64BestEffort({end: String}, 3, 'Asia/Shanghai')
GROUP BY ship_serial_no, bucket_start
ORDER BY time ASC, shipId ASC;

流程优化设计

首屏加载优化

  1. 应用启动时先同步预热默认实时窗口的最新船位缓存。
  2. 全库统计异步预热，不阻塞最新船位缓存准备。
  3. 页面初始化先请求 /api/config/map，建图后立即执行 loadLatest() 请求最新船位并渲染 marker。
  4. 前端通过 deferAuxiliaryWork()、统计开关和摘要调度器再请求统计、热力格数等辅助信息。

轨迹加载优化

环节 设计 收益
抽稀 写入链路维护固定抽稀点，读请求不在原始表在线分桶聚合。 读请求不在原始表在线分桶聚合。
框选 候选船查询先走细网格索引，再做 bbox 相交。 避免从原始点找每艘候选船。

统计与分析隔离

  • 数据库总量统计从 RealtimeService 缓存取值，不在每次实时船位刷新时扫描全库。
  • 热力密度从 300 秒抽稀表聚合，且要求当前视野 bbox。
  • 态势分析时间步进由前端拆成多个小窗口请求，后端每次只处理一个静态时间片。
  • 统计请求有本地缓存、请求中断和开关控制；关闭统计后不再请求统计接口。
  • API trace 默认不记录 /api/stats/* 和配置接口，避免统计链路干扰主链路观测。
  • 请求指标和日志记录只保留接口级耗时、数据库调用量和汇总字段，不进入轨迹点级循环。

前端页面设计

页面结构

左侧模式导航

  • 实时位置
  • 态势分析
  • 单船轨迹
  • 多船轨迹
  • 全域回放

主工作区

  • 地图与图层：底图、轨迹线、回放轨迹、框选矩形、船位 marker、热力。
  • 右侧面板：统计指标、时间输入、模式操作控件、播放控制。
  • 多船候选抽屉：类型过滤、分页、批量选择、加载轨迹。

各模式交互

模式 输入控件 主动作 渲染结果
实时位置 时间点、前推分钟、信号源类型、已选船。 刷新全部最新船位；点击船可预览一小时轨迹。 最新船位 marker、悬浮信息、视野船数。
态势分析 开始时间、结束时间、步进分钟、当前地图视野。 按当前视野生成热力。 时间演进密度热力与热力统计。
单船轨迹 船舶编号、时间点、前后小时、抖动抑制、抽稀模式。 查询单船轨迹并播放。 单船轨迹线、当前位置、进度条。
多船轨迹 时间范围、框选范围、候选类型、候选分页与选择。 框选后加载候选，再加载已选船轨迹。 播放窗口内多船轨迹与 marker。
全域回放 时间点、向前小时、抽稀模式。 加载全域片段。 当前播放帧的全域船位。

前端性能设计

  • switchMode() 切换模式时清理轨迹播放状态和图层，避免旧图层叠加。
  • renderTracks() 先准备分组与播放运行时，再由批处理追加线要素。
  • 多船播放只绘制当前播放时刻向前约 30 分钟轨迹窗口，防止多船全线常驻。
  • 全域播放只刷新位置 marker，不绘制全域轨迹线。
  • 性能模式下单船播放走增量 marker/轨迹同步，减少重复重建。
  • 候选船按批次和页面加载，抽屉内保存已加载页面与选择状态。
  • 统计面板由 stats-toggle 控制；关闭时中断未完成统计请求并清空统计摘要状态。
  • 统计接口使用 getCachedJson() 短缓存，重复切换或刷新时优先复用已有统计结果。
  • 实时模式低 zoom 热力通过 scheduleRealtimeHeat() 延后触发，不阻塞 marker 首屏绘制。
  • 接口耗时浮层只跟踪主业务接口，统计接口不进入前端 API trace 历史。

约束与后续建议

当前约束

  • 固定抽稀粒度只有 60、300、1800 秒，新增粒度需要新增或调整写入侧维护方案。
  • 候选船索引按 30 分钟桶近似时间范围，首末时间是桶时间，不等于原始首末事件时间。
  • 多船轨迹当前在选船后不按原框选 bbox 截断，前端语义应明确为“所选船时间窗轨迹”。
  • 单船原始模式可能显著放大返回量，需保持显式入口，不应成为默认值。

建议验收项

  1. 分别验证实时首屏缓存命中、缓存未命中、ClickHouse 异常时的响应行为。
  2. 用 EXPLAIN indexes = 1 对比候选船索引查询与原始表窗口统计读取范围。
  3. 以典型 zoom 级别验证抽稀桶选择、返回点数、前端帧率。
  4. 压测多船候选分页和多船轨迹加载上限，确认不影响实时船位刷新。
